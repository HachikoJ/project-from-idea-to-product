# Phase 1: Project Definition

## 这一阶段的目标

不要急着做页面。  
这一阶段只解决一件事：

> 把一个模糊想法，变成一个边界清楚、值得推进的项目。

如果这一步没做好，后面很容易出现：
- 做着做着发现需求变了
- 页面越做越多，但主链路不清
- AI 输出很多内容，但没有真正可执行的方向

---

## 必问的 7 个问题

### 1. 这个项目到底解决什么问题？
不要回答“做一个系统”。
要回答：
- 它替谁解决什么麻烦
- 这个麻烦现在是怎么被低效处理的

### 2. 谁在用？
至少区分：
- 提交者 / 普通用户
- 管理者 / 处理者
- 是否还有第三角色（审核者、协作者、老板）

### 3. 核心场景是什么？
要写出真实动作链，而不是抽象名词。
例如：
- 提交工单
- 分配处理
- 更新进度
- 通知用户
- 关闭工单

### 4. 主链路是什么？
项目第一版必须先跑通哪条链？
默认写成：
用户动作 → 系统响应 → 管理动作 → 用户反馈结果

### 5. 第一版必须成立什么？
只保留必要项。
不要把“以后可能想要的功能”混进 MVP。

### 6. 暂时不做什么？
必须明确排除项，例如：
- 先不做导出
- 先不做复杂权限
- 先不做数据分析面板
- 先不做自动化规则引擎

### 7. 这个项目最终要做到什么交付形态？
至少确认：
- 只是内部使用
- 还是要公开展示
- 是否要部署到真实域名
- 是否要写教程 / GitHub / docs 页面

## 建议输出格式

### 项目一句话定义
用一句话说清：
“这是一个给谁用、解决什么问题、通过什么主链路完成价值交付的系统。”

---

## 成功标准

这一阶段完成，不是“想法更多了”，而是：
- 项目说得清
- 角色分得清
- 主链路定得下
- MVP 收得住
- 后续可以正式进入拆解阶段
