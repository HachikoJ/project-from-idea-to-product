# Phase 5: Surgical Debugging

## 核心原则

外科手术式纠偏的目标不是“顺手优化一堆地方”，而是：

- 只修当前问题
- 尽量不动 unrelated 模块
- 保持现有结构稳定
- 用最小改动恢复正确行为

---

## 默认排查顺序

### 1. 先定义现象
不要直接改代码，先说清：
- 用户看到什么
- 预期应该是什么
- 差异发生在哪一步

### 2. 再判断问题层级
优先判断属于哪一类：
- 路径问题
- 接口问题
- 状态依赖问题
- 数据预加载问题
- 布局问题
- 交互流程问题

### 3. 只修根因，不修表象
例如：
- 通知点击打不开详情，不要先改按钮样式
- 线上 API 报错，不要先怀疑数据库
- 页面挤乱，不要一开始重写整块布局

---

## 默认修复模板

1. 当前问题：一句话说清楚
2. 根因判断：为什么会发生
3. 最小改动策略：只改哪一处
4. 明确不改什么：避免越修越多
5. 验收标准：修好后应该出现什么结果

---

## 常见高频纠偏模式

### 模式 1：路径前缀漏掉
常见于：
- 子路径部署
- 前后端分离后上线
- 某个接口改漏

默认检查：
- 前端 fetch 路径
- 静态资源路径
- basePath / assetPrefix
- Nginx rewrite / proxy

### 模式 2：逻辑依赖预加载页面
常见于：
- 通知中心点击详情
- 列表页跳详情页
- 某个功能只有先打开 A 页后 B 页才正常

默认原则：
> 页面跳转或详情打开必须尽量 direct-open，不要依赖另一个页面先加载本地状态。

### 模式 3：高密度区布局混乱
常见于：
- 标题 + 操作按钮挤在一行
- 按钮被压缩换行
- 时间信息和操作信息打架

默认修法：
- 把时间拆到独立行
- 按钮强制不换行
- 让标题承担换行
- 不轻易重写整卡片结构

### 模式 4：移动端浏览器行为问题
常见于：
- 输入框缩放
- fixed 区域抖动
- 键盘弹起导致错位

默认原则：
- 先确认是不是浏览器行为
- 再决定是 CSS、viewport 还是交互时机问题

---

## 禁止事项

- 不要把纠偏任务顺手扩成重构任务
- 不要在用户只要求修一个问题时顺手改一堆 UI
- 不要因为现象复杂就直接全盘推翻
- 不要只看代码，不看真实使用路径

---

## 成功标准

外科手术式纠偏成功，不是“代码更漂亮了”，而是：
- 目标问题消失
- 其他模块没被误伤
- 验收结果明确
- 修改范围可控
