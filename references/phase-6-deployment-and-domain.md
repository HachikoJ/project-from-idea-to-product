# Phase 6: Deployment And Domain

## 为什么部署到自己域名是项目的一部分

一个项目如果只停在本地，它更像练习。  
当它跑到你自己的域名上，它才真正具备：

- 真实访问入口
- 可展示性
- 可分享性
- 可交付性
- 成果掌控感

所以部署不是“最后顺手一做”，而是产品化的一部分。

---

## 默认检查框架

### 1. 先搞清部署形态
先判断项目是：
- 根路径部署
- 子路径部署
- 前后端同域
- 前后端反向代理汇总到同一域名

不同部署形态，路径处理策略完全不同。

### 2. 再统一路径意识
重点看：
- 页面访问路径
- API 请求路径
- 静态资源路径
- 构建时路径设置
- Nginx rewrite / proxy 行为

### 3. 最后看进程与稳定性
重点看：
- 服务是否常驻
- 重启后是否自动拉起
- 错误时是否容易定位
- 日志是否可追

---

## 子路径部署特别提醒

如果项目不是挂在 `/`，而是挂在例如：
- `/anonyproof`
- `/ticket-system`
- `/internal-tools`

那就必须统一考虑：
- 前端 base path
- 资源前缀
- API 前缀
- 反向代理规则

默认原则：
> 只要跑在子路径下，就要把“路径一致性”当成一等公民。

---

## 上线排坑默认顺序

1. 页面能不能打开
2. 静态资源是不是 200
3. API 能不能通
4. 用户行为链路能不能走通
5. 刷新页面会不会炸
6. 管理端和用户端路径是否一致

---

## 关于“部署到自己域名”的表达原则

如果项目需要对外展示或用于教学、销售、交付，应该强调：

- 用户对成果拥有真正掌控权
- 成果不再停留在 localhost
- 项目会从练习品变成作品
- 域名部署会显著提升成就感和可信度

但表达要自然，不要写成廉价营销口号。

更好的表达方式是：
> 当一个项目真正跑在你自己购买的域名上时，它才会第一次像一个属于你的产品。

---

## 成功标准

部署阶段完成，不是“服务起来了”，而是：
- 页面访问正常
- API 行为正常
- 关键链路正常
- 路径统一
- 域名访问稳定
- 用户可以真实打开和演示
