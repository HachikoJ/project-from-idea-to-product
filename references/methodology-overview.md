# Methodology Overview

## 这个方法论解决的根问题

很多项目做不出来，不是因为不会写代码，而是因为：

- 想法没有被拆成系统
- 没有优先级
- 一上来就堆功能
- AI 跑偏后只会重开
- 本地跑通后就误以为项目结束了
- 最后没有上线、没有包装、没有交付形态

这个方法论的目标，是把一个模糊想法推进成一个真实成果。

---

## 总流程

### 1. 项目定义
先回答：
- 这个系统到底解决什么问题
- 谁在用
- 谁在管理
- 为什么值得做
- 第一版必须成立什么

### 2. 需求拆解
把一句话需求拆成：
- 页面
- 模块
- 状态流转
- 角色关系
- MVP 闭环

### 3. 产品与交互定义
不要只问“页面怎么写”，要先定义：
- 首页建立什么认知
- 列表页承接什么动作
- 详情页最重要的信息是什么
- 高密度区如何控复杂度

### 4. 开发推进
默认顺序：
1. 打通最小闭环
2. 补关键状态流转
3. 补通知 / 评论 / 搜索 / 筛选
4. 修移动端
5. 做上线部署
6. 做文档与包装

### 5. 外科手术式纠偏
遇到问题时，不要反复重开。
先判断：
- 是路径问题
- 是接口问题
- 是状态问题
- 是布局问题
- 还是流程假设本身错了

只修最小范围。

### 6. 上线与交付
默认完成标准不是“本地能跑”，而是：
- 可以上线
- 可以访问
- 可以演示
- 可以交付
- 可以复盘

---

## 这套方法最适合什么项目

- 内部工单系统
- 审批系统
- 客服 / 售后后台
- 反馈系统
- 小团队业务工具
- 双端后台类产品
- 需要部署到自己域名的工具型项目

---

## 这套方法最强调的 5 件事

1. 先拆系统，不先写页面
2. 先做闭环，不先堆功能
3. 先判断阶段，再决定动作
4. 先纠偏，再考虑重构
5. 先做成成果，再谈优化
