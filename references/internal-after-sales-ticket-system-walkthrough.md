# Internal After-Sales Ticket System Walkthrough

## 这个 reference 的作用

这个文件不是新 skill 的主体逻辑，而是一个**完整试跑样本**。

它用“内部售后工单系统”这个题目，演示 `project-from-idea-to-product` 这套方法是如何从一个想法，一路推进到：

- 项目定义
- 需求拆解
- 产品与交互推进
- 开发顺序安排
- 外科手术式纠偏思路
- 上线部署形态
- 包装与交付方向

它的意义不是固定这个业务场景，而是证明：

> 这套方法可以从《匿证》迁移到另一个全新项目，并仍然保持结构化推进能力。

---

## Phase 1：项目定义

### 一句话定义
这是一个给团队内部统一收集、分配、跟进、闭环售后问题的系统，目标是避免请求散落在微信、群聊、私聊和口头沟通里，导致漏单、拖延、责任不清和进度不可见。

### 角色
- 提交人：销售、客服、运营、内部同事
- 处理人：售后人员、技术支持、产品支持
- 管理员：主管、负责人

### 主链路
提交工单 → 管理员查看 → 分配处理人 → 处理人更新状态与回复 → 提交人查看进度并补充 → 工单关闭

### MVP
- 创建工单
- 查看我的工单
- 工单详情
- 管理端工单列表
- 分配处理人
- 更新状态
- 评论补充

### 暂不做
- 复杂权限矩阵
- 智能派单
- 高级报表
- 多级审批
- 大而全通知系统

### MVP 闭环

## Phase 3：产品与交互推进

### 用户端定位
不是管理台，而是：
- 快速提单
- 查看进度
- 补充信息
- 确认结果

### 用户端首页重点
- 新建工单
- 我的工单概览
- 最近动态 / 通知

### 列表页重点
- 状态最醒目
- 标题其次
- “待我补充”要明显
- 避免堆过多字段

### 详情页重点
详情页是闭环中心，应突出：
- 工单摘要
- 当前状态
- 沟通记录
- 下一步由谁动作

### 管理端重点
- 快速看积压
- 快速看责任人
- 快速定位高优先级问题
- 具体推进动作收敛到处理页

## Phase 4：开发推进顺序

### 第 1 阶段：最小闭环
先做：
- 创建工单
- 我的工单列表
- 工单详情
- 管理端工单列表
- 分配处理人
- 状态更新

### 第 2 阶段：沟通补全
再做：
- 评论 / 回复
- 补充信息
- 待补充状态
- 最近更新时间

### 第 3 阶段：通知与轻量管理效率
再做：
- 站内通知
- 未读提醒
- 通知直达详情
- 状态 / 优先级 / 分配筛选

### 第 4 阶段：体验优化
再做：
- 用户端移动端适配
- 信息层级优化
- 高密度区布局整理

### 第 5 阶段：上线与包装
最后做：
- 部署
- 域名 / 内网入口
- README
- GitHub
- 教程
- docs

## Phase 5：外科手术式纠偏

### 常见偏差
1. 主链路没通，先堆辅助功能
2. 详情页变成信息垃圾桶
3. 通知只是展示，不是动作入口
4. 管理端列表越做越重
5. 本地正常，线上路径混乱

### 默认纠偏法
- 先定义现象
- 再判断问题层级
- 找根因
- 只做最小改动
- 写清验收标准

### 典型原则
- 不轻易重构
- 不顺手改 unrelated 模块
- 不把局部 bug 扩成全局重写

## Phase 6：上线部署与域名形态

### 推荐形态
优先按“可挂在正式域名子路径下”的结构设计，例如：
- `/tickets`
- `/support`
- `/after-sales`

### 推荐部署结构
- 前端：Next.js / React
- 后端：Express / Node
- 进程管理：PM2
- 反向代理：Nginx

### 关键意识
如果项目跑在子路径下，必须统一处理：
- 页面路径
- API 路径
- 静态资源路径
- 反向代理规则

### 判断标准
本地跑通 ≠ 成果成立。  
只有跑到真实访问入口后，项目才开始具备真正系统形态。
