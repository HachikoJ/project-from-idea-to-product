# Anonyproof Case Study

## 为什么《匿证》值得作为案例

《匿证》不是单页 demo，而是一个完整的小产品：

- 用户端
- 管理端
- 通知系统
- 评论沟通
- 状态流转
- 移动端
- 上线部署
- 文档包装
- GitHub 备份

这使它很适合被抽象成项目推进样本。

---

## 《匿证》项目里最关键的可复用经验

### 1. 不要把一句话需求直接交给 AI
“做一个匿名反馈系统”太模糊。
真正有效的做法是先拆出：
- 用户端
- 管理端
- 通知闭环
- 状态流转
- 评论
- 部署路径
- 文档展示

### 2. 主链路必须先打通
《匿证》的关键主链路是：
用户提交反馈 → 管理员看到 → 管理员处理 → 用户收到通知 → 用户查看详情 → 用户继续评论

先打通这条链，再补优化模块。

### 3. 子路径部署是高频真实坑
《匿证》部署在 `/anonyproof` 子路径下。
因此必须统一处理：
- `basePath`
- `assetPrefix`
- 前端 API 地址
- Nginx 反向代理路径

这类坑在“本地没问题、线上全出错”的项目里非常常见。

### 4. 通知系统不能依赖页面预加载
真实踩坑：
- 通知能看到
- 但用户必须先进入“我的反馈”，再回通知页，才能打开详情

根因：
- 点击逻辑错误依赖本地 `myFeedbacks`

可复用原则：
> 通知跳转必须具备 direct-open 能力，不能依赖另一个页面先被打开。

### 5. UI 产品感来自约束，不来自形容词
比如通知中心：
- 标题可换行
- 按钮不换行
- 时间单独一行
- 高密度信息区要克制

这比“做高级一点”有效得多。

### 6. 移动端问题不是最后再看
《匿证》真实出现过移动端输入缩放问题。
这说明：
- 移动端体验不是锦上添花
- 浏览器行为也会直接影响用户对产品质感的判断

### 7. 文档和截图属于产品的一部分
《匿证》后期有大量工作不是写功能，而是：
- README
- docs 页
- 截图映射
- 图片命名
- 教程稿
- GitHub 对外展示

这说明：
> 能跑，不等于能交付；能交付，也不等于能卖相。

## 《匿证》可抽象成哪些通用模式

- 双角色产品（用户 / 管理者）
- 状态流转产品
- 通知驱动闭环产品
- 子路径部署项目
- 先做功能，再做包装的产品化项目

---

## 哪些部分不能被生搬硬套

不能直接复用的包括：
- 匿名反馈这个业务场景本身
- 页面名称
- 角色命名
- 状态文案
- 所有功能结构

真正应该复用的，是：
- 拆解方式
- 推进顺序
- 纠偏方法
- 上线路径意识
- 文档包装方式
