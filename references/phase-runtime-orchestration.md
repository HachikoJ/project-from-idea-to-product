# Phase Runtime Orchestration

## 这个文件解决什么问题

前面的 references 解决的是“怎么想清楚项目”。
这个文件解决的是：

> 项目一旦开始推进，当前到底该做什么，谁该停，谁该进，什么时候进入下一阶段。

它是这个 skill 的运行指挥层。

---

## 默认运行规则

### 1. 永远先判断当前阶段
不要上来就生成一堆任务。
先判断项目当前处于：
- 项目定义期
- 需求拆解期
- 主链路搭建期
- 协作补全期
- 通知 / 管理增强期
- 体验修正期
- 上线部署期
- 包装交付期

### 2. 每次只输出一个当前最高优先级任务
不要一次并行输出 5 个方向。
默认输出：
- 当前最值得做的 1 个任务
- 当前明确不做的 2~5 个事项
- 当前任务完成后的验收标准

### 3. 每个阶段都要有停止线
如果没有停止线，项目就会一直在当前阶段打转。

例如：
- 主链路阶段的停止线：一张工单能从创建走到状态更新
- 包装阶段的停止线：README、GitHub、基础教程已完成

### 4. 没达到退出条件，不进入下一阶段
不要因为“看起来差不多”就跳阶段。
必须根据可验证标准判断。

---

## 默认输出格式

### 当前阶段
一句话说明现在在哪个阶段。

### 当前唯一高优先级任务
只保留一个。

### 当前不要做什么
列 2~5 条，防止发散。

### 当前任务验收标准
必须可验证，不能写成空话。

### 完成后下一步
只给一个下一步方向。

---

## 阶段切换判断规则

### 从项目定义期 → 需求拆解期
前提：
- 用户是谁清楚
- 角色清楚
- 主链路清楚
- MVP 收住

### 从需求拆解期 → 主链路搭建期
前提：
- 页面清单清楚
- 模块清楚
- 状态流转清楚
- 第一阶段只做什么已经明确

### 从主链路搭建期 → 协作补全期
前提：
- 提交 / 列表 / 详情 / 分配 / 状态更新跑通

### 从协作补全期 → 管理增强期
前提：
- 评论补充链路成立
- 工单能真实推进，而不是静态展示

### 从管理增强期 → 体验修正期
前提：
- 轻量通知、筛选、基础管理效率已成立

### 从体验修正期 → 上线部署期
前提：
- 主链路稳定
- 关键页面可接受
- 没有明显阻塞型 bug

### 从上线部署期 → 包装交付期
前提：
- 真实访问入口可用
- 线上关键链路通过

---

## 最重要的原则

这个 skill 不是任务列表生成器，而是项目节奏控制器。

它必须持续帮助用户和龙虾判断：
- 现在该往前推进
- 还是该停下来补闭环
- 还是该收手进入下一阶段
