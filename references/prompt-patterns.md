# Prompt Patterns

## 1. 项目定义提示词

```text
我现在不是要做一个 demo，而是要做一个真实可推进、可上线、可交付的项目。
请先不要写代码，先帮我把这个项目定义清楚：
1. 目标用户是谁
2. 谁是管理者/处理者
3. 核心场景是什么
4. 主链路是什么
5. 第一版必须成立什么
6. 暂时不做什么
输出要结构化，不要空话。
```

---

## 2. 需求拆解提示词

```text
请把这个项目从一句话需求，拆成真实可执行的项目结构。
至少输出：
- 页面清单
- 功能模块
- 角色关系
- 状态流转
- MVP 闭环
- 推荐开发顺序
要求优先突出主链路，不要一上来堆可选功能。
```

---

## 3. 上帝视角提示词

```text
不要把当前任务当成单页开发，而要站在整个产品闭环角度思考。
这个项目要同时考虑：
- 用户端体验
- 管理端流程
- 通知/状态闭环
- 移动端适配
- 上线部署
- GitHub/文档/交付形态
当前优先级不是堆功能，而是先把主链路做通。
```

---

## 4. UI / UX 提示词

```text
不要只追求“好看”，要优先保证信息层级、阅读秩序和操作确定性。
当前页面属于高密度信息区，请按以下约束优化：
- 标题可以换行
- 操作按钮必须保持单行
- 时间信息独立一行
- 按钮不能被压缩
- 风格克制、可信、产品感强
- 不要像博客模板或练习作品
```

---

## 5. 外科手术式纠偏提示词

```text
不要重构整页，只修这一个具体问题。
当前问题不是整体逻辑错误，而是：[写清具体错误点]
请：
1. 保持现有结构和样式不变
2. 只修复这一处逻辑
3. 不要顺手修改 unrelated 模块
4. 修复完成标准是：[写清验收结果]
如果需要新增兜底逻辑，请使用最小改动方案。
```

---

## 6. 上线部署提示词

```text
请从真实上线角度检查这个项目，而不是只按本地开发视角判断。
重点检查：
- 页面路径
- API 路径
- 子路径部署约束
- 静态资源引用
- 反向代理
- 进程管理
- 域名访问形态
输出时请按“最容易出错的地方 → 根因 → 修复建议”结构说明。
```

---

## 7. 文档包装提示词

```text
请不要把它写成开发日志，而要写成真实产品/真实教程文档。
优先保证：
- 用户能看懂
- 图文一致
- 页面角色清楚
- 电脑版与移动版区分明确
- 文档像产品交付，不像过程记录
```

---

## 8. 复盘抽象提示词

```text
请不要只总结这个项目本身，而要把它抽象成下次还能复用的方法。
输出时区分：
1. 这次项目特有的部分
2. 未来项目可复用的方法
3. 以后应该沉淀进 skill 的原则
```
